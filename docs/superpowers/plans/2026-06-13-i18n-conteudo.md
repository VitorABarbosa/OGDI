# i18n Conteúdo (Plano 2) — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Extrair toda a prosa PT hardcoded das páginas (seções + data files) para os catálogos next-intl e traduzi-la para EN e ES, página a página, seguindo o glossário de marca.

**Architecture:** A infra do Plano 1 já existe (next-intl, `app/[locale]`, provider, `messages/{pt,en,es}.json` com `nav`/`common`/`footer`). Este plano adiciona **um namespace por página** (`home.*`, `institucional.*`, ...) aos três catálogos. Cada componente de seção passa a ler seus textos via `getTranslations` (server) ou `useTranslations` (client); os `*.data.ts` mantêm estrutura/ids e trocam prosa por **chaves de tradução**. As traduções EN/ES são produzidas seguindo `docs/superpowers/specs/2026-06-13-i18n-glossario.md` (autoridade de conteúdo).

**Tech Stack:** Next.js 15.5.19 (App Router), next-intl v4, TypeScript.

**Autoridade de conteúdo:** `docs/superpowers/specs/2026-06-13-i18n-glossario.md`. Decisões travadas: `operação`→**venture** (EN); nav ES `institucional` permanece "Institucional"; ES = **neutro LatAm formal**; prosa provisória dos `[slug]` fica **PT** (só UI traduzida).

## Procedimento de extração e tradução (compartilhado por todas as tasks de página)

Cada task de página aplica ESTE procedimento. Está definido uma vez aqui (DRY); cada task abaixo só informa seus arquivos, o nome do namespace e suas particularidades.

**P1. Inventariar as strings.** Ler cada componente de seção e cada data file da página. Listar toda string PT voltada ao usuário: textos entre tags JSX (`>texto<`), props de texto (`title=`, `kicker=`, `aria-label=`, `alt=` de conteúdo, `placeholder=`), e prosa em `*.data.ts`. **NÃO** extrair: classes CSS, ids, hrefs, nomes próprios da lista "não traduzir" do glossário (Open Group, OGDI, nomes de empreendimentos/empresas/pessoas/lugares, CEF, SCP), números/datas.

**P2. Definir as chaves.** Para o namespace da página (ex.: `home`), criar sub-chaves por seção e por papel semântico: ex. `home.hero.kicker`, `home.atuacao.title`, `home.atuacao.steps.01.title`. Para arrays em data files, usar o **id/idx estável já existente** como segmento de chave (ex. `idx: "01"` → `home.atuacao.steps.01`). Chaves em inglês, kebab/camel conforme já usado em `nav`/`common`.

**P3. Popular `messages/pt.json`.** Adicionar o namespace com os textos PT **verbatim** (copiar exatamente o que estava no código — é a fonte da verdade PT).

**P4. Religar os componentes.**
- **Server Component** (sem `"use client"`): tornar `async` se necessário e usar `const t = await getTranslations("<namespace>")`; trocar cada literal por `t("chave")`. Para listas vindas de data file, iterar o array (que agora tem `key`/`idx`) e renderizar `t("...".concat(item.idx))`.
- **Client Component** (`"use client"`): `const t = useTranslations("<namespace>")`; idem.
- Trechos com ênfase inline (ex. manifesto com `<em>`) usam `t.rich("chave", { em: (c) => <em>{c}</em> })` com a marcação `<em>...</em>` embutida no valor da mensagem.
- Substituir `Link`/`a` de navegação interna por `Link` de `@/i18n/navigation` **apenas se o componente fizer navegação interna e ainda usar `next/link`** (mantém o locale). Não trocar âncoras externas.

**P5. Ajustar data files.** Em `*.data.ts`, remover os campos de prosa (title/desc/kick/etc.) e manter estrutura (id/idx/href/flags/tone/image). Onde o componente precisar, derivar o texto por `t(...)` usando o id/idx. Se um data file vira só ids+chaves, simplificar o tipo. Manter comentários técnicos.

**P6. Traduzir EN e ES.** Adicionar o MESMO namespace a `messages/en.json` e `messages/es.json`, com tradução de cada chave seguindo o glossário (tom, termos-âncora, lista de proibidos). Paridade de chaves obrigatória entre os 3 arquivos.

**P7. Verificar.**
- `npx tsc --noEmit` limpo.
- `npm run build` — sem erro de **missing message key** (next-intl falha o build se uma chave usada não existir nos 3 locales).
- Smoke da página nos 3 locales (procedimento no fim do plano, seção "Smoke por página"): confirmar que o conteúdo aparece traduzido em `/en/<rota>` e `/es/<rota>` e PT em `/<rota>`.

**P8. Commit** com mensagem `feat(i18n): traduz pagina <rota> (namespace <ns>)`.

> **Paridade de chaves — checagem rápida** (rodar quando útil):
> ```bash
> node -e "const fs=require('fs');const flat=(o,p='')=>Object.entries(o).flatMap(([k,v])=>v&&typeof v==='object'?flat(v,p+k+'.'):[p+k]);const L=['pt','en','es'].map(l=>({l,k:flat(JSON.parse(fs.readFileSync('messages/'+l+'.json','utf8'))).sort()}));const b=L[0].k.join();L.forEach(x=>console.log(x.l,x.k.length,x.k.join()===b?'OK':'MISMATCH'))"
> ```
